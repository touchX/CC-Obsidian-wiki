## 项目概览

### 核心实验

在过去的五个月里，OpenAI 团队进行了一项实验：**构建并交付一款软件产品的内部 beta 版，其中没有一行代码是人工编写的**。

**关键数据**：
- ⏱️ **开发时间**：约 5 个月（2025 年 8 月下旬开始）
- 📊 **代码规模**：约一百万行代码
- 🔄 **Pull Requests**：约 1,500 个 PR
- 👥 **团队规模**：3 名工程师（初期）→ 7 名工程师（现在）
- 📈 **吞吐量**：平均每位工程师每天处理 3.5 个 PR
- 🚀 **效率提升**：约手工编写代码所需时间的 1/10

**产品状态**：
- ✅ 有内部日常活跃用户
- ✅ 有外部 Alpha 测试者
- ✅ 经历了交付、部署、故障和修复的完整过程

### 核心理念

> **人类掌舵。智能体执行。**

这一限制是有意选择的，目的是构建必要的内容，从而将工程速度提升数个数量级。

**工程角色的转变**：
- ❌ **不再是**：编写代码
- ✅ **而是**：设计环境、明确意图、构建反馈回路

## 从零开始的旅程

### 空的 Git 仓库

首次提交到一个空的代码仓库是在 **2025 年 8 月下旬**。

**初始架构**（由 Codex CLI 使用 GPT-5 生成）：
- 代码仓库结构
- CI 配置
- 格式化规则
- 包管理器设置
- 应用框架
- **AGENTS.md** 文件（指导智能体如何在代码仓库中工作）

**关键点**：
- 系统没有预存任何人工编写的代码
- 从一开始，代码仓库就由智能体塑造
- 就连指导智能体的初始 AGENTS.md 文件本身也是由 Codex 编写的

### 团队核心理念

**不手动编写代码** — 这成为团队的核心理念。

## 工程师角色的重新定义

由于缺乏人工编码的实践，**工程师工作的重点转向了系统、架构和杠杆作用**。

### 深度优先的工作方式

早期进展比预期的慢，这并不是因为 Codex 不具备相应的能力，而是因为**环境的规范不够明确**。

**解决方案**：
1. 将更大的目标拆解为更小的构建模块（设计、代码、评审、测试等）
2. 提示智能体去构建这些模块
3. 使用它们去解锁更复杂的任务

**关键思维转变**：
- ❌ "再努力一点"
- ✅ "究竟还需要什么样的能力，我们又该如何让这个能力对智能体来说既清晰可读又可强制执行？"

### Ralph Wiggum 循环

人类几乎完全通过提示与系统交互：

1. 工程师描述任务
2. 运行智能体
3. 允许其打开一个 Pull Request
4. 指示 Codex 在本地审核其自身的更改
5. 在本地和云端请求额外的特定智能体审查
6. 对任何人工或智能体给出的反馈做出响应
7. 循环往复，直到所有智能体审核人员都满意为止

**工具使用**：
- Codex 直接使用标准开发工具（gh、本地脚本、嵌入代码仓库的技能）
- 无需人工将内容复制粘贴到 CLI 中

### 审核

- ✅ 人类可以审核 Pull Request
- ❌ 但并非必须这样做
- 🔄 随着时间推移，已将几乎所有的审核工作调整为**智能体对智能体**的方式来处理

## 提高应用程序的可读性

### 瓶颈转移

随着代码吞吐量的增加，瓶颈变成了**人工 QA 能力**。

由于人类的时间和注意力是固定的限制因素，团队一直在努力通过令应用程序的 UI、日志和应用指标等内容对 Codex 直接可读，从而为智能体增加更多功能。

### Git Worktree

令应用程序可以根据 git worktree 启动，因此 Codex 可以为**每次更改启动并驱动一个实例**。

### Chrome DevTools 协议

将 Chrome DevTools 协议接入智能体运行时，并创建了用于处理以下内容的技能：
- DOM 快照
- 屏幕截图
- 导航

**能力提升**：
- ✅ 复现错误
- ✅ 验证修复
- ✅ 直接推理 UI 的行为

**工作流程**：
1. 选择一个目标
2. 对触发用户界面路径前后的状态进行快照
3. 通过 Chrome DevTools 观察运行时事件
4. 应用修复
5. 重启
6. 循环重新运行验证，直到应用程序恢复正常

### 可观测性堆栈

日志、指标和追踪记录会通过一个本地可观测性堆栈展示给 Codex：
- **Vector**：数据分发
- **Victoria Logs**：日志查询（LogQL）
- **Victoria Metrics**：指标查询（PromQL）
- **Victoria Traces**：追踪查询（TraceQL）

**关键特性**：
- 对任何给定的工作树来说，该堆栈都是临时的
- Codex 在应用程序的完全独立的版本上运行
- 一旦任务完成，该版本的所有内容（包括日志和指标）都会被删除

**实际效果**：
- 单次 Codex 运行在单个任务上持续工作超过**六个小时**（通常在人类睡眠时间）
- 提示变得可行："确保服务启动在 800ms 内完成"或"这四个关键用户旅程中的任何跨度都不得超过两秒"

## 代码仓库作为记录系统

### 情境管理挑战

**情境管理**是使智能体在大型和复杂任务中有效发挥作用的最大挑战之一。

**早期经验教训**：
> **要给 Codex 的是一张地图，而不是一本 1,000 页的说明书。**

### "大型 AGENTS.md" 的失败

尝试了"一个大型的 `AGENTS.md`"方法，但这是一次失败的尝试：

| 问题 | 说明 |
|------|------|
| **情境是一种稀缺资源** | 巨大的指令文件会挤掉任务、代码和相关文档 — 智能体要么会错过关键约束条件，要么开始针对错误的约束条件进行优化 |
| **过多的指导反而变得无效** | 当一切都"重要"时，一切都不重要了。智能体最终会在本地进行模式匹配，而不是有意识地进行导航 |
| **它会立即腐烂** | 庞杂的手册会变成陈旧规则的坟场。智能体无法判断哪些信息仍然有效 |
| **这很难核实** | 单个 blob 不适合进行机械检查（覆盖率、新鲜度、所有权、交叉链接），因此漂移是不可避免的 |

### 解决方案：内容目录

不再将 `AGENTS.md` 视为百科全书，而是将其视为**内容目录**。

**代码仓库知识库结构**：

```
AGENTS.md
ARCHITECTURE.md
docs/
├── design-docs/
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/
│   └── db-schema.md
├── product-specs/
│   ├── index.md
│   ├── new-user-onboarding.md
│   └── ...
├── references/
│   ├── design-system-reference-llms.txt
│   ├── nixpacks-llms.txt
│   ├── uv-llms.txt
│   └── ...
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

**关键特性**：
- 📁 **结构化的 `docs/` 目录**：被当作记录系统来使用
- 🗺️ **简短的 `AGENTS.md`**（约 100 行）：主要用作地图，指向其他地方更深层次的真实信息来源
- ✅ **渐进式披露**：智能体从小而稳定的切入点开始，并被指导下一步该去哪里查看

**严格执行**：
- 专职的 linter 和 CI 作业会验证知识库的更新状况、是否已交叉链接且结构正确
- 定期运行的"doc-gardening"智能体会扫描过时或废弃文档，并发起修复用的 Pull Request

## 目标是智能体的可读性

### 为 Codex 优化

由于代码库完全由智能体生成，因此**首先针对 Codex 的可读性进行了优化**。

**核心原则**：
> 就像团队会努力提升代码对新入职工程师的可导航性一样，人类工程师的目标也是让智能体能够**直接从代码仓库**推理出完整的业务领域。

### 知识的局限性

从智能体的角度来看，**它在运行时无法在情境中访问的任何内容都是不存在的**。

**不可见的知识**：
- ❌ Google Docs
- ❌ 聊天记录（Slack 消息）
- ❌ 人们头脑中的知识

**可见的知识**：
- ✅ 代码仓库本地的、已版本化的工件（代码、Markdown、模式、可执行计划）

**推送到仓库**：
> 那次让团队在架构模式上达成一致的 Slack 讨论？如果智能体无法发现它，那么它就会像迟了三个月入职的新员工一样，对其一无所知。

### 选择可推理的依赖

为 Codex 提供更多情境意味着要组织和展示正确的信息，好令智能体能够基于这些信息进行推理，而不是用临时指令使其不堪重负。

**决策框架**：
- 倾向于选择那些可以**完全内化于在仓库中进行推理**的依赖项和抽象
- 对智能体来说，"枯燥"的技术往往更容易建立模型（可组合性、API 稳定性、训练集表现）
- 在某些情况下，让智能体重新实现部分功能子集比绕过公共库中不透明的上游行为更便宜

**示例**：
> 我们没有引入通用的 `p-limit` 风格包，而是投入使用了我们自己的带并发的 map 辅助函数：它与我们的 OpenTelemetry 仪表紧密集成，具备 100% 的测试覆盖率，并且其行为完全符合我们的运行时预期。

## 规范架构与品味

### 不变量优于微观管理

> **通过强制执行不变量，而非对实施过程进行微观管理，我们令智能体能够快速交付，而且不会削弱基础。**

**示例**：
- ✅ 要求 Codex 在边界处解析数据形状（parse, don't validate）
- ❌ 不规定具体实现方式（模型似乎偏好 Zod，但没有指定特定库）

### 严格的架构模型

智能体在具有**严格边界和可预测结构**的环境中最为高效。

**架构规则**：
- 每个业务域划分为一组固定的层
- 依赖方向经过严格验证
- 仅允许有限的一组边
- 横切关注点（认证、连接器、遥测、功能标志）通过单一的显式接口进入：Providers

**依赖方向**：
```
Types → Config → Repo → Service → Runtime → UI
```

**强制执行**：
- 自定义的 linter（由 Codex 生成）
- 结构测试
- 机械地强制执行约束

**时机**：
> 这种架构通常要等到你拥有数百名工程师时才会推迟。对于编码智能体来说，这是一个**早期的先决条件**：有了约束，速度才不会下降，架构才不会漂移。

### 品味不变式

通过自定义 lint 静态地强制执行：
- 结构化日志记录
- 模式和类型的命名约定
- 文件大小限制
- 特定平台的可靠性要求

**关键点**：
- 这些 lint 是自定义的，错误信息会在智能体情境中注入修复指令
- 在以人为本的工作流程中可能会让人感到迂腐或束缚
- 有了智能体，它们就成了**倍增器**：一旦编码，就能立即应用于所有地方

**中央 vs 本地**：
- **中央层面**：强制执行边界、正确性和可重复性
- **本地层面**：允许团队或智能体在解决方案的表达方式上拥有很大的自由

### 代码风格

生成的代码**不总是符合人类的风格偏好**，这也没关系。

**达标标准**：
- ✅ 输出是正确的
- ✅ 可维护的
- ✅ 对未来的智能体运行而言清晰易读

## 吞吐量改变了合并的理念

随着 Codex 的吞吐量增加，许多传统的工程规范变得不再有效。

### 减少阻塞合并门

- Pull Request 的生命周期很短
- 测试偶发失败通常通过后续重跑来解决，而不是无限期地阻碍进展
- 在智能体吞吐量远超人类注意力的系统中，**纠错成本低，而等待成本高**

**权衡**：
> 在低吞吐量环境中，这样做是不负责任的。而在这里，这通常是正确的选择。

## "智能体生成"的含义

### 完整的生成范围

当说代码库是由 Codex 智能体生成的，指的是**整个代码库**。

**智能体的产出**：
- ✅ 产品代码与测试
- ✅ CI 配置和发布工具
- ✅ 内部开发者工具
- ✅ 文档和设计历史
- ✅ 评估框架
- ✅ 审阅评论和回复
- ✅ 管理代码仓库本身的脚本
- ✅ 生产仪表板定义文件

### 人类的角色

人类始终参与其中，但工作的抽象层次与过去不同：
- 优先处理工作
- 将用户反馈转化为验收标准
- 对结果进行验证

**反馈回路**：
> 当智能体遇到困难时，我们将其视为一个信号：识别缺失的内容 — 工具、指导与约束、文档 — 并将其反馈到代码仓库中，始终由 Codex 自己编写修复。

### 不断提高的自主水平

随着越来越多的开发环节被直接编码到系统中（测试、验证、审查、反馈处理和恢复），代码仓库最近跨过了一个重要门槛：**Codex 能够端到端地驱动一个新功能**。

**端到端能力**：
1. 验证代码库的当前状态
2. 重现已报告的漏洞
3. 录制一个演示故障的视频
4. 实施修复措施
5. 通过运行应用程序来验证修复
6. 录制第二个视频，演示解决方案
7. 打开 Pull Request
8. 回应智能体和人类反馈
9. 检测并修复构建故障
10. 仅在需要判断时才交由人工处理
11. 合并更改

**注意**：
> 此行为在很大程度上取决于此代码仓库的具体结构和工具，不应在没有类似投入的情况下假定它可以泛化 — 至少目前还不行。

## 熵与垃圾收集

### 漂移问题

**完全自主的智能体也引入了新的问题。** Codex 会复现代码仓库中已存在的模式 — 甚至包括那些不均衡或不够理想的模式。

随着时间的推移，这不可避免地导致**漂移**。

### 早期解决方案

最初，人类手动处理这个问题：
- 每周五（占一周的 20%）花时间清理"AI 残渣"
- ❌ 不具备可扩展性

### 黄金原则 + 垃圾回收

开始将"黄金原则"直接编码到代码仓库中，并建立了一个**循环清理流程**。

**黄金原则**：带有主观意见的机械规则，旨在保持代码库的可读性和一致性，以便将来运行智能体。

**示例**：
1. 更倾向于使用共享的实用程序包，而不是手工编写的辅助工具（以便将不变式集中管理）
2. 不会使用"YOLO 式"探测数据 — 会验证边界，或依赖类型化的 SDK

**后台任务**：
- 定期运行一组后台 Codex 任务
- 扫描偏差、更新质量等级
- 发起有针对性的重构 Pull Request
- 大多数可以在一分钟内完成审查并自动合并

**类比**：
> 其功能类似于垃圾回收。技术债务就像一笔高息贷款：不断地以小额贷款的方式偿还债务，总比让债务不断累积，再痛苦地一次解决要好得多。

**效果**：
- 人类的品味一旦被捕捉，就会持续应用于每一行代码
- 每天发现并解决不良模式
- 而不是让它们在代码库中传播数天或数周

## 仍在学习的内容

### 当前状态

到目前为止，这一策略在 OpenAI 的内部发布和采纳过程中表现良好：
- 为真实用户打造真实产品
- 帮助将投资锚定在现实中
- 引导实现长期的可维护性

### 未解决的问题

1. **架构连贯性**：在一个完全由智能体生成的系统中，架构连贯性会如何随着时间的推移而演变
2. **人类判断力**：仍在学习人类的判断力在哪些方面能发挥最大作用，以及如何对这种判断力进行编码
3. **系统演变**：随着时间的推移，模型的功能不断增强，这一系统将如何演变

### 关键洞察

> 显而易见的是：构建软件仍然需要纪律，但纪律更多地体现在**支撑结构**上，而不是代码上。

> 保持代码库一致性的工具、抽象和反馈回路变得越发重要。

**最棘手的挑战**：
- 设计环境
- 反馈回路
- 控制系统

帮助智能体实现目标：**大规模构建和维护复杂、可靠的软件**。
