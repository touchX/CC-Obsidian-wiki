## 核心痛点与解决方案

### 三大痛点

使用 Claude Code 干复杂任务时，经常遇到以下问题：

| 痛点 | 问题表现 | 根本原因 |
|------|----------|----------|
| **上下文膨胀** | 对话越来越长，开始遗忘前面修改的内容 | 单 Agent 上下文有限 |
| **任务串行** | 能并行的任务却只能一个一个串行执行 | 缺乏并行执行机制 |
| **视角单一** | 一个 Agent 只顾安全，顾不上性能 | 单 Agent 结构瓶颈 |

### Agent Teams 解决方案

Claude 官方发布的 **Agent Teams** 功能解决了这些痛点：

> [!tip] 核心架构
> 一个 **Team Lead**（团队领导）拆解任务，多个 **Teammate**（团队成员）并行执行，还能互相通信协调。

## 什么是 Agent Teams

### 基本定义

Agent Teams 允许多个 Claude Code 实例协同工作：

```
用户（你）
    ↓ 发送任务
Team Lead（团队领导）
    ↓ 拆解任务
    ├── Team Member 1 → 后端开发
    ├── Team Member 2 → 前端开发
    └── Team Member 3 → QA 测试
```

### 核心特性

| 特性 | 说明 |
|------|------|
| **多 Agent 协同** | 多个 Claude Code 实例并行工作 |
| **任务分发** | Team Lead 拆解任务，派发给成员 |
| **互相通信** | 团队成员之间可以互相通信 |
| **状态共享** | 共享任务列表，实时更新状态 |

---

## Agent Teams vs SubAgent

### 核心区别

| 维度 | SubAgent | Agent Teams |
|------|----------|------------|
| **通信方式** | 只能向主人报告结果 | 成员之间可以互相通信 |
| **协作模式** | 主从模式 | 对等协作模式 |
| **适用场景** | 专注工作者，快速完成任务 | 需要分享、发现、互相质疑、自主协调 |

### 选择指南

**选择 SubAgent**：
- 需要快速专注地工作者
- 任务可以独立完成
- 只需要报告结果

**选择 Agent Teams**：
- 需要多个 Agent 互相讨论
- 需要互相质疑对方的发现
- 需要自主协调完成任务

### 实战选择题

> **题目 1**：在 5 个目录中搜索特定模式的文件
> **答案**：SubAgent ✅（典型串行场景，不需要跨绘画协调）

> **题目 2**：做 PR 审查，希望多个审查者互相质疑对方的发现
> **答案**：Agent Teams ✅（需要协作和讨论）

## 如何启用 Agent Teams

### 方法一：配置文件

在 `~/.claude/settings.json` 中添加：

```json
{
  "experimental": {
    "agentTeams": true
  }
}
```

### 方法二：临时环境变量

```bash
export CLAUDE_AGENT_TEAMS_ENABLED=true
```

### 启用后

重启 Claude Code 终端即可生效。

## 触发团队协作

### 基本语法

```bash
# 让 Claude 创建 Agent Teams
"创建一个 5 人的 Agent Teams，包含：架构设计、后端、前端、QA、Review"
```

### 示例：代码审查

```markdown
请创建一个代码审查团队，审核这个 PR。
团队需要三个角色：
- 专注安全审查
- 检查性能问题
- 检查测试覆盖率
```

### 执行流程

1. Team Lead 创建审查团队
2. 启动多个 Agent（并行）
3. 每个 Agent 独立执行任务
4. 成员之间可以通信
5. 结果汇报给 Team Lead

## 架构与通信机制

### 核心架构

```
┌─────────────────────────────────────────────────────────┐
│                    Team Lead (领导)                     │
│  • 创建主会话                                            │
│  • 派生团队成员                                          │
│  • 分配任务和依赖                                        │
│  • 协调通信                                              │
└─────────────────────────────────────────────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ↓                 ↓                 ↓
┌───────────────┐ ┌───────────────┐ ┌───────────────┐
│ Team Member 1 │ │ Team Member 2 │ │ Team Member 3 │
│  (独立实例)   │ │  (独立实例)   │ │  (独立实例)   │
│  • 收到提示词 │ │  • 收到提示词 │ │  • 收到提示词 │
│  • 领取任务   │ │  • 领取任务   │ │  • 领取任务   │
│  • 执行并汇报 │ │  • 执行并汇报 │ │  • 执行并汇报 │
└───────────────┘ └───────────────┘ └───────────────┘
        ↑                 ↑                 ↑
        └─────────────────┴─────────────────┘
                         通信
```

### 关键组件

| 组件 | 功能 |
|------|------|
| **任务列表** | Team Lead 创建，指定任务依赖关系 |
| **消息系统** | Agent 之间直接通信 |
| **文件锁机制** | 防止多个成员同时认领同一任务 |

### 任务状态

| 状态 | 说明 |
|------|------|
| **待处理 (pending)** | 等待成员领取 |
| **进行中 (in_progress)** | 成员正在执行 |
| **已完成 (completed)** | 任务完成 |

### 权限继承

团队成员的权限继承自 Team Lead：
- 如果给领导最高权限 → 成员也继承最高权限

## 真实项目实战流程

### 项目场景

创建一个完整的 Web 应用功能：

```
架构设计 → 后端开发（并行）→ 前端开发（并行）→ QA 测试 → Code Review
```

### 执行步骤

#### 1. 初始化团队

```bash
"创建一个 5 人的 Agent Teams：
 - 架构设计
 - 后端开发
 - 前端开发
 - QA
 - Code Review"
```

#### 2. 任务分配

Team Lead 执行：
1. 创建任务
2. 设置依赖关系（设计先行）
3. 分配给团队成员

#### 3. 并行开发

- 后端和前端**并行**执行
- 每个 Agent 有独立的 Claude Code 实例
- 接收各自的提示词和任务

#### 4. 协调执行

```
后端/前端完成 ──→ QA 接手测试 ──→ Code Review 进行审查
```

每个阶段由 Team Lead 协调。

### 性能数据

| 指标 | 数值 |
|------|------|
| **Token 消耗** | ~30M tokens |
| **完成度** | ~80% |
| **Bug 数量** | 2-3 个（前端+后端各有一些） |

## 最佳使用场景

### 1. 研究与审查

多个 Agent 从不同角度分析问题，互相质疑。

### 2. 新模块开发

前后端并行开发，利用任务依赖管理协调。

### 3. 竞争性假设

发现问题后，多个 Agent 互相讨论：
- "我反对你的发现"
- "你也反对我的发现"
- 最终得出真正可能的答案

### 4. 跨层协调

传统开发流程中，前后端可以并行：
```
架构设计 ──→ 后端 + 前端（并行）──→ QA ──→ Review
```

## 总结

**Agent Teams 的价值**：
- ✅ 解决单 Agent 的结构性瓶颈

- ✅ 适合复杂任务的协作完成

> 单 Agent 只能串行工作 → Team Lead + Teammate = 并行协作
