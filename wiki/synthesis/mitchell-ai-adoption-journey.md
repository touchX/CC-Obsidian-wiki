## 核心心智模型

### 三阶段采用曲线

Mitchell 的经验：采用任何有意义的工具都需要经过三个阶段：

1. **低效期**（Inefficiency）— 感觉比手工更慢
2. **适应期**（Adequacy）— 开始找到价值
3. **突破期**（Discovery）— 工作流和生活的改变

**关键洞察**：
> 在大多数情况下，我必须强迫自己通过第 1 和第 2 阶段，因为我通常有一个我已经满意和舒适的工作流程。采用工具感觉像工作，我不想付出努力，但我通常会为了成为我手艺的全面人才而努力。

### 务实态度

- 在过度炒作、夸张的观点海洋中
- 代表对 AI 的更细致、衡量的方法
- 不在乎 AI 是否会留在这里
- 只是想要构建东西的软件工匠

## 第一步：放弃 Chatbot

### 核心建议

**立即停止尝试通过聊天机器人（如 ChatGPT、Gemini Web）执行有意义的工作。**

**为什么**：
- Chatbot 有真正的价值，是 AI 工作流的日常部分
- 但在编码中的效用非常有限
- 你主要是希望它们根据先前的训练得出正确结果
- 纠正它们涉及人类（你）反复告诉它们错误
- 这是低效的

### 使用 Agent 代替

**Agent 定义**：可以聊天并在循环中调用外部行为的 LLM

**最低要求**：
- ✅ 读取文件
- ✅ 执行程序
- ✅ 发起 HTTP 请求

**个人经历**：
- 第一个"哇"时刻：将 Zed 命令面板截图粘贴到 Gemini
- 要求它用 SwiftUI 复制
- 几秒钟内完成，效果非常好
- 但在其他任务中令人失望
- 在棕地项目中，聊天界面经常产生糟糕的结果

### 现代编码模型

**Opus 和 Codex** 等现代编码模型经过专门训练，与对话模型相比，倾向于使用工具。

## 第二步：重现你自己的工作

### 强制复制策略

Mitchell 尝试了 **Claude Code**，最初并不印象深刻：

> 我只是没有从会话中得到好的结果。我觉得我必须润色它产生的所有东西，这个过程比我自己做需要更多时间。

**但他没有放弃**，而是：
> **强迫自己用 agent 重现所有手动提交。**
> 我确实做了两次工作。我手动做工作，然后与 agent 斗争以产生质量和功能相同的结果（当然，它不能看到我的手动解决方案）。

### 为什么这样做

**这是极其痛苦的**，因为它阻碍了简单地完成任务。但：
- 摩擦是自然的
- 如果不耗尽努力，就无法得出坚定、可辩护的结论
- 从第一原则发现他人已经说过的东西
- 发现自己导致更强的基本理解

### 从实践中发现的关键原则

1. **将会话分解为单独的清晰、可操作的任务**
   - 不要尝试在一个 mega 会话中"画猫头鹰"

2. **对于模糊的请求，将工作分成单独的规划和执行会话**

3. **如果你给 agent 一种验证其工作的方法，它通常会修复自己的错误并防止回归**

### 效率提升

- 导致显著的效率提升
- 开始自然地使用 agent，感觉不比自己做慢
- 但仍不觉得更快，因为主要是在"看管"agent

**负面空间的价值**：
- 效率提升的部分是了解**何时不使用** agent
- 对于可能失败的任务使用 agent 显然是浪费时间
- 拥有避免这种情况的知识导致时间节省

**在此阶段**：
- 对 agent 感到满意，愿意在工作流中使用它们
- 但仍未感到任何净效率提升
- 对此并不在乎，对 AI 作为工具感到满意

## 第三步：日终 Agent

### 新模式

**尝试获得一些效率**：**每天的最后 30 分钟启动一个或多个 agent。**

**假设**：也许如果 agent 在我无法工作的时间取得一些积极进展，我可以获得一些效率。

基本想法：不是尝试在我拥有的时间内做更多，而是尝试在我不拥有的时间内做更多。

### 起初的挫折

- 最初发现既不成功又烦人
- 但很快发现真正有帮助的不同类别的工作

### 有效的任务类型

#### 1. 深度研究会话

要求 agent 调查某个领域：
- 查找特定语言中具有特定许可证类型的所有库
- 为每个库生成多页摘要
- 优点、缺点、开发活动、社会情绪等

#### 2. 并行 Agent 尝试不同的想法

- 我有一些模糊的想法但没有时间开始
- 不期望它们生产我会发布的东西
- 但可能在第二天我做任务时照亮一些未知的未知

#### 3. Issue 和 PR 分类/审查

- Agent 擅长使用 `gh`（GitHub CLI）
- 手动编写快速方法并行启动一堆进行 issue 分类
- **不允许 agent 响应**，只是想要第二天报告
- 帮助指导我走向高价值或低努力任务

### 实践细节

- 没有像其他人那样让 agent 整夜循环运行
- 大多数情况下，agent 在不到半小时内完成任务
- 工作日的后半部分，通常疲劳，退出心流
- 发现个人效率低下
- 将努力转移到启动这些 agent 给了我第二天早上的"热启动"
- 比否则更快地开始工作

**在此阶段**：
- 很高兴
- 开始感觉比 AI 之前做得更多（即使只是轻微）

---

## 第四步：外包稳赢任务

### 核心策略

**让 agent 做所有它们擅长的工作，同时我做其他任务。**

### 每日流程

1. **每天开始**：取前一天晚上分类 agent 的结果
2. **手动过滤**：找到 agent 几乎肯定能很好解决的问题
3. **后台运行**：一个接一个（不是并行）地让它们继续
4. **同时**：**我做其他事情**

**关键**：
- 不去社交媒体（比没有 AI 时更多）
- 不看视频等
- 在我自己的、正常的、AI 前的深度思考模式
- 工作在我想要或必须工作上

### 关键实践

**关闭 agent 桌面通知**：
- 上下文切换非常昂贵
- 为了保持效率，人类的工作是控制何时中断 agent
- 而不是反过来
- **不要让 agent 通知你**
- 在工作的自然休息期间，切换并检查它，然后继续

### 技能形成的权衡

这有助于对抗高度宣传的 [Anthropic 技能形成论文](https://www.anthropic.com/research/AI-assistance-coding-skills)：

- 权衡：你没有为你委托给 agent 的任务形成技能
- 同时继续在你继续手动工作的任务中自然形成技能

**在此阶段**：
- 坚定地处于"无法回头"的领域
- 感觉更有效率
- 即使不是，最喜欢的部分是现在可以专注于编码和思考我真正喜欢的任务
- 同时仍然充分完成我不喜欢的任务

## 第五步：工程化 Harness

### 核心理念

**显而易见**：当 agent 第一次产生正确结果，或最坏情况产生需要最小润色的结果时，agent 效率要高得多。

实现这一点的最可靠方法是给 agent 快速、高质量的工具，自动告诉它何时错误。

### Harness Engineering

Mitchell 称之为 **"Harness Engineering"**：
> 任何时候你发现 agent 犯了错误，你花时间工程化一个解决方案，使 agent 永远不会再犯那个错误。

### 两种形式

#### 1. 更好的隐式提示（AGENTS.md）

对于简单的事情：
- agent 反复运行错误的命令
- 找到错误的 API
- 更新 `AGENTS.md`（或等效文件）

**Ghostty 示例**：
[AGENTS.md](https://github.com/ghostty-org/ghostty/blob/ca07f8c3f775fe437d46722db80a755c2b6e6399/src/inspector/AGENTS.md)
- 该文件中的每一行都基于不良的 agent 行为
- 几乎完全解决了它们所有

#### 2. 实际的编程工具

- 脚本来截图
- 运行过滤测试
- 等等
- 通常与 AGENTS.md 更改配对，让它知道这个现有工具

### 当前状态

**这就是 Mitchell 今天所处的阶段。**

- 每当看到 agent 做 Bad Thing 时，真诚努力防止它再做那个坏事
- 或者，真诚努力让 agent 能够验证它们在做 Good Thing

## 第六步：始终让 Agent 运行

### 目标

**与第五步同时**，Mitchell 还在以下目标下运作：**始终让 agent 运行。**

- 如果 agent 没有运行，问自己"agent 现在能为我做什么？"

### 偏好模型

特别喜欢将此与较慢、更深思熟虑的模型结合：
- Amp 的 [deep mode](https://ampcode.com/news/deep-mode)（基本上就是 GPT-5.2-Codex）
- 可能需要 30+ 分钟才能做出小改动
- 另一方面，它确实倾向于产生非常好的结果

### 当前实践

- **[还？]不运行多个 agent**，目前也不太想
- 发现拥有一个 agent 运行是很好的平衡
- 在能够做我喜欢的深度手动工作和看管我那种有点愚蠢但神秘地多产的机器人朋友之间

### 目标 vs 现实

"始终让 agent 运行"仍然只是一个目标：
- 现在可能在一个正常工作日的 10-20% 有效地让后台 agent 运行
- 但正在积极努力改善这一点

**重要态度**：
> 我不想为了运行 agent 而运行 agent。
> 我只想在我认为真正对我有帮助的任务上运行它们。
> 这个目标的挑战部分是改进我自己的工作流和工具，以便我可以有一系列持续的高质量工作要委托。
> 即使没有 AI，这也很重要！

## 今天的状态

### 旅程成果

通过这个旅程，Mitchell 个人达到了一个点：
- 现代化 AI 工具取得成功
- 相信自己以基于现实的适当衡量方法接近它
- 不在乎 AI 是否会留在这里
- 只是想为了对游戏的热爱而构建东西的软件工匠

### 诚实态度

整个格局发展如此迅速：
- 相信很快就会回顾这篇文章并嘲笑自己的天真
- 但如果你不能为过去的自己感到尴尬，你可能没有成长
- 只希望朝着正确的方向成长

- 只是想分享个人方法来导航这些新工具
- 给出关于**如何一般地接近新工具**的一瞥，无论 AI

## 关键洞察

### 六步旅程

| 步骤 | 核心行动 | 关键洞察 |
|------|---------|---------|
| **1. 放弃 Chatbot** | 使用 Agent 而非聊天界面 | Agent = LLM + 工具调用能力 |
| **2. 重现工作** | 强制自己做两次工作 | 从实践中发现原则，而非阅读 |
| **3. 日终 Agent** | 利用无法工作的时间 | 研究任务、并行尝试、分类审查 |
| **4. 外包稳赢** | Agent 做擅长的事，你做其他事 | 关闭通知，防止上下文切换 |
| **5. 工程 Harness** | 防止 Agent 犯重复错误 | AGENTS.md + 编程工具 |
| **6. 始终运行** | 保持后台 Agent 持续工作 | 不为了运行而运行，只做有价值的事 |

### 实践原则

1. **务实胜过炒作** — 从怀疑开始，逐步验证
2. **强制实践** — 通过做两次工作来发现模式
3. **渐进采用** — 从低效 → 适应 → 突破
4. **防止错误** — 工程化 Harness，而非重复纠正
5. **持续运行** - 始终让 Agent 在后台工作
6. **技能平衡** — 委托任务但保留核心技能

### 与其他概念的关系

- **Harness Engineering** — Martin Fowler 的理论框架
- **Agent = Model + Harness** — LangChain 的定义
- **长期运行 Agent** — Anthropic 的实践
- **Codex Harness** — OpenAI 的完全 AI 编写项目
