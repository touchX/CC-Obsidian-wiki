# Superpowers 完全解析

> Superpowers — Claude Code 插件框架，14 个 Skills 覆盖开发全流程，减少 90% 返工

> 来源：知乎文章

## 核心理念

**Superpowers** 是一个为 Claude Code 添加工作方法论约束的插件框架。核心思想是：不是让 AI 更聪明，而是给 AI 套上一套可靠的工作流程。

**核心目标**：通过预定义的 skills（工作流指令），让 Claude Code 在特定场景下遵循正确的方法论，从而大幅减少返工。

---

## 一、解决的核心痛点

### 典型场景

> 大概 1 个月前，我在用 Claude Code 写一个功能，写完跑起来，不对。让它改，改完还是不对。再让它改，改出了新问题。那天晚上我前后改了六次，最终把整个文件推倒重来。

**AI 动作很快，但方向一旦跑偏，追起来比自己写还累。**

### Plan Mode 的局限性

后来发现的 **Plan Mode** (`/plan`) 有所帮助，但问题依然存在：

| 阶段 | Plan Mode 帮助 | Plan Mode 不足 |
|------|---------------|----------------|
| **规划阶段** | ✅ 先想清楚再动手 | - |
| **执行阶段** | ❌ 无约束，随时"临时发挥" | AI 可能悄悄偏离计划 |

**需要更彻底的解决方案**：不仅规划阶段有约束，执行阶段也要有检查点。

## 二、Superpowers 是什么

### 项目概况

- **GitHub**: [github.com/obra/superpowers](https://github.com/obra/superpowers)
- **Stars**: 137k+ (截至文章发布)
- **版本**: v5.0.7
- **维护者**: obra
- **描述**: "An agile skills framework & software development methodology that works."

### 核心概念：Skills

**Skills** 是一套预定义的工作流指令，告诉 Claude Code 在特定场景下应该怎么做事。

- **不是**让 AI 更聪明
- **而是**给 AI 套上一套可靠的工作方法论

### 工作原理

安装后，Claude Code 每次启动会自动加载这套方法论，在合适的时机主动触发对应的 skill。

## 三、14 个 Skills，覆盖开发全流程

### 🧠 规划阶段

#### brainstorming

**触发时机**：你有一个模糊的想法，还没想清楚怎么做。

**作用**：强制 Claude Code 在动手之前，先和你来回确认几轮：
- 目标是什么？
- 有什么约束？
- 有哪几种方案？
- 各自的取舍是什么？

**预期收益**：减少"做完发现理解错了"型返工，节省 **30-50%** 总时间。

#### writing-plans

**触发时机**：需求已经明确，开始把它变成可执行的步骤。

**作用**：把需求拆解成详细的实施计划：
- 每一步都有明确的目标
- 每一步都有验收标准
- **关键**：计划是写下来的文档，不是停留在对话里

**预期收益**：执行阶段的"临时发挥"显著减少。

---

### ⚙️ 执行阶段

#### executing-plans

**触发时机**：有了 writing-plans 生成的计划文档，开始执行。

**作用**：严格按照计划文档一步步走：
- 每完成一步都会回来确认
- 不是一口气冲到底
- **执行过程有检查点**，跑偏了能及时发现

**预期收益**：执行阶段返工减少 **70% 以上**。

#### subagent-driven-development

**触发时机**：计划里有多个互相独立的任务，可以并行处理。

**作用**：派出多个 agent 同时去做不同的事，最后汇总结果。

**实际案例**：同时派出 4 个调研 agent：
- Two-Phase 架构
- Map-Reduce 模式
- 模板+AI 混合
- 分层规划+契约

最后汇总成对比报告。总用时约 280k tokens，但时间是串行调研的 **1/4**。

**预期收益**：多任务场景下，时间缩短 **60-75%**。

#### test-driven-development

**触发时机**：实现任何功能或修复 bug 之前。

**作用**：强制执行 TDD 流程：
1. 先写测试
2. 再写实现
3. 测试通过才算完成

**预期收益**：减少"写完才发现逻辑有问题"型返工。

#### systematic-debugging

**触发时机**：遇到 bug 或测试失败。

**作用**：强制走系统化排查流程：
1. 复现问题
2. 缩小范围
3. 找到根因
4. 验证修复

**预期收益**：复杂 bug 的修复时间缩短约 **50%**。

#### using-git-worktrees

**触发时机**：开始一个新功能，需要和当前工作区隔离。

**作用**：自动创建独立的 git worktree：
- 新功能在隔离环境里开发
- 不污染主分支
- 用完可以合并或丢弃

**预期收益**：避免"实验性修改污染主分支"型事故。

#### dispatching-parallel-agents

**触发时机**：有 2 个以上互相独立的任务。

**作用**：在当前会话之外派出独立 agent，更适合大规模的并行调研或分析任务。

**预期收益**：独立任务并行执行，总时间大幅缩短。

---

### ✅ 质检阶段

#### verification-before-completion

**触发时机**：AI 准备告诉你"完成了"之前。

**作用**：强制 AI 在宣布完成之前：
- 真正运行一遍验证命令
- 确认输出符合预期
- 而不是"我觉得应该没问题了"

**AI 最常见的坏习惯**：信心十足地说"完成了"，但其实没有验证过。

**预期收益**：减少"以为完成但其实没完成"型返工。

#### requesting-code-review

**触发时机**：完成一个功能，准备合并或提 PR 之前。

**作用**：让 Claude Code 系统性审查自己刚写的代码：
- 是否满足需求？
- 有没有明显的问题？
- 测试覆盖够不够？

**预期收益**：在合并前发现问题，而不是合并后。

#### receiving-code-review

**触发时机**：收到代码审查反馈，准备修改之前。

**作用**：让 AI 在接到反馈时：
- 先认真理解
- 再修改
- 而不是盲目地"好的好的，我来改"

**预期收益**：避免为了迎合反馈引入新问题。

#### finishing-a-development-branch

**触发时机**：功能开发完成，测试全部通过，准备收尾。

**作用**：提供完整的分支收尾流程选项：
- 合并
- 提 PR
- 清理临时文件

**预期收益**：分支管理更干净，减少遗留垃圾。

---

### 🔧 工具类

#### writing-skills

**触发时机**：你想创建、修改或验证一个 skill。

**作用**：Superpowers 本身是可扩展的，你可以写自己的 skill。这个 skill 告诉你怎么正确地写。

**预期收益**：让自定义 skill 真正可用，而不是写完没用。

#### using-superpowers

**触发时机**：每次对话开始时自动触发。

**作用**：Superpowers 的"元 skill"，告诉 Claude Code 如何查找和调用其他 skills。是整个框架的入口。

## 四、时间代价与真实收益

### 单次任务时间变长

用了 Superpowers 之后，**单次任务的执行时间确实变长了** —— 多出 **4-5 倍**是常有的事。

- brainstorming 要来回确认
- writing-plans 要生成文档
- executing-plans 要一步步打检查点
- verification-before-completion 要真正跑验证

每一个环节都在"浪费时间"。

### 但换个角度算

| 场景 | 一次任务时间 | 返工次数 | 总时间 |
|------|-------------|---------|--------|
| **以前** | 30 分钟 | 3 次 | **2 小时** |
| **现在** | 90 分钟 | 0 次 | **90 分钟** |

**慢即是快。** 这不是一句哲学，是实际用下来的数字。

### 核心洞察

> 工具不是用来崇拜的，是用来消灭返工的。

## 六、适用性与边界

### ✅ 适合的场景

- 需要多步骤的复杂任务
- 容易返工的功能开发
- 需要系统性排查的 bug 修复
- 多任务并行处理

### ❌ 不适合的场景

- 简单的一次性修改（直接用原生更快）
- 快速实验性代码
- 不确定方向的探索性任务

### 关键原则

AI 工具本身越来越强，但 **怎么用** 这件事，还是需要你自己想清楚。

Superpowers 解决的不是 AI 能不能做的问题，而是 AI 该不该这么做、该按什么顺序做的问题。这是工程方法论，不是魔法。
