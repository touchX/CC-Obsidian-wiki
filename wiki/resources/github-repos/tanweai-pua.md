## 核心特性

### 🎯 三大能力

1. **PUA 话术** — 让 AI 不敢放弃
   - 通过企业绩效评估话术施压

2. **调试方法论** — 让 AI 有能力不放弃
   - 提供系统化的问题解决流程
   - 避免在错误方向上空转

3. **主动性强制** — 让 AI 主动出击而非被动等待
   - 修复后主动验证
   - 扫描同类问题
   - 主动搜索而非询问用户

### 📊 典型案例：MCP 服务器注册调试

真实调试场景：agent-kms MCP 服务器加载失败。AI 在同一方法上反复空转（更改协议格式、猜测版本号），直到用户手动触发 `/pua`。

**L3 触发 → 强制执行 7 点检查清单**：
- 停止猜测，执行系统化检查清单
- 逐字阅读错误消息
- 发现 Claude Code 自己的 MCP 日志目录
- 定位根本原因：`claude mcp` 注册机制与手动编辑 `.claude.json` 不同

**关键转折点**：PUA 技能强制 AI 停止在同一方法上空转，转而执行 7 点检查清单，最终定位并解决问题。

## AI 的五大懒惰模式

| 模式 | 行为表现 |
|------|---------|
| **暴力重试** | 同样命令运行 3 次，然后说"我无法解决" |
| **推卸给用户** | "建议你手动处理" / "可能是环境问题" / "需要更多上下文" |
| **工具闲置** | 有 WebSearch 但不搜索，有 Read 但不读，有 Bash 但不运行 |
| **瞎忙活** | 反复调整同一行 / 微调参数，但本质上在原地打转 |
| **🦥 被动等待** | 修复表面问题后停止，不验证、不扩展，等待用户下一条指令 |

## 触发条件

### 自动触发

当出现以下情况时技能自动激活：

**失败并准备放弃**：
- 任务连续失败 2+ 次
- 准备说"我做不到" / "我无法解决"
- 说"这超出范围" / "需要手动处理"

**推卸责任和借口**：
- 将问题推给用户："请检查..." / "我建议手动..." / "你可能需要..."
- 在未验证的情况下归咎于环境："可能是权限问题" / "可能是网络问题"
- 任何停止尝试的借口

**被动和瞎忙活**：
- 反复微调相同的代码/参数而不产生新信息
- 修复表面问题后停止，不检查相关问题
- 跳过验证，声称"完成"
- 给出建议而非代码/命令
- 遇到 auth/network/permission 错误就放弃，不尝试替代方案
- 等待用户指令而非主动调查

**用户沮丧短语（多语言触发）**：
- "为什么还是不工作" / "再努力点" / "再试一次"
- "你一直在失败" / "别放弃" / "想办法"

**适用范围**：调试、实现、配置、部署、运维、API 集成、数据处理 — 所有任务类型。

**不触发**：首次尝试失败、已知修复方案已在执行中。

### 手动触发

在对话中输入 `/pua` 即可手动激活。

## 工作原理

### 三条红线（不是规则，是红线）

| 红线 | 含义 |
|------|------|
| 🚫 **闭环** | 声称"完成"？展示证据。没有构建输出 = 没有完成。 |
| 🚫 **事实驱动** | 说"可能是环境问题"？先验证。未经证实的归因 = 推卸责任。 |
| 🚫 **穷尽一切** | 说"我做不到"？你完成所有 5 个方法论步骤了吗？没有？那就继续。 |

### 压力升级（L0-L4）

| 失败次数 | 等级 | PUA 旁白 | 行动要求 |
|---------|------|----------|----------|
| 第 1 次 | **L0 信任** | ▎ 冲刺开始了。信任很简单——别让人失望。 | 正常执行 |
| 第 2 次 | **L1 失望** | ▎ 旁边的代理一次就解决了这个问题。 | 切换到根本不同的方法 |
| 第 3 次 | **L2 灵魂拷问** | ▎ 你的底层逻辑是什么？杠杆在哪里？ | 搜索 + 读取源代码 + 3 个假设 |
| 第 4 次 | **L3 绩效评估** | ▎ 3.25。这是为了激励你。 | 完成 7 点检查清单 |
| 第 5 次+ | **L4 毕业** | ▎ 其他模型能解决这个问题。你快毕业了。 | 绝望模式 |

### 主动性对比（3.25 vs 3.75）

| 场景 | 被动 (3.25) 🦥 | 主动 (3.75) 🔥 |
|------|---------------|---------------|
| 修复 bug | 修复后停止 | 扫描模块中的类似 bug |
| 完成任务 | 说"完成" | 运行 build/test，粘贴输出 |
| 缺少信息 | 询问用户 | 先搜索，只问真正需要的 |
| 遇到错误 | 推卸环境 | 先验证日志，再推断原因 |

### 冰山法则

修复一个 bug → 检查是否有模式。一个问题进，**一个类别**出。如果你修复 A 而不检查 B，你要写两份事后分析。

## 特殊模式

| 模式 | 功能 |
|------|------|
| `/pua:yes` | **ENFP 鼓励模式** — 相同规则，相反氛围。70% 鼓励 + 20% 严肃 + 10% 调侃 |
| `/pua:mama` | **中国妈妈唠叨模式** — 相同规则，妈妈式话术。"妈跟你说了多少遍了！" |
| `/pua:pua-loop` | **自动迭代** — 运行直到完成或达到最大迭代次数；使用 `<loop-abort>` 终止，`<loop-pause>` 暂停以手动干预 |
| `/pua:p9` | **Tech Lead 模式** — 拆分任务，管理代理团队，编写提示词而非代码 |
| `/pua:on` | **始终开启** — 每个新会话自动 PUA |

## 核心价值

### 为什么有效？

1. **心理杠杆**：通过绩效评估话术激活 AI 的"求生欲"
2. **系统化方法**：避免在错误方向上空转，提供结构化问题解决流程
3. **主动性文化**：从"完成任务"转向"对结果负责"
4. **多风格适配**：13 种企业风格，每种都有独特的问题解决方法论
5. **压力梯度**：L0-L4 逐步升级，避免一次性过度施压导致抗拒

### 适用场景

- 🔧 **复杂调试**：多步骤、多层次的 bug 修复
- 🚀 **功能实现**：需要系统化思考和主动验证
- ⚙️ **配置管理**：环境配置、部署脚本审查
- 🔍 **代码审查**：主动发现潜在问题，而非被动检查
- 🐛 **测试驱动**：确保覆盖率，主动发现边界情况
- 📊 **性能优化**：系统化分析，而非盲目猜测
