## 架构概览

天气系统是一个完整的工作示例，演示了在一个编排流程中运行的 **两种不同的技能模式** 。 命令作为入口点，将数据获取委托给一个携带预加载领域知识的子 Agent，然后调用一个独立技能进行可视化渲染。这种分离并非偶然 —— 它阐明了 Claude Code 扩展模型的一个基本设计原则： **当 Agent 需要持久的领域上下文时预加载技能，当需要一次性能力时独立调用技能** 。

## 深入解析：命令作为编排器

命令是此工作流的架构骨干。它使用 —— 刻意选择最轻量的模型 —— 因为它的工作是 **协调，而非认知** 。它提出一个问题，进行两次工具调用，并格式化一个摘要。将复杂操作（API 调用、SVG 生成）委托给专门的子组件既更可靠，也更具成本效益。

### Frontmatter 配置

限制是命令的主要执行机制。通过仅允许三个工具，该命令在架构上根本无法绕过 Agent 或技能 —— 它不能直接调用 ，不能写入文件，也不能执行 bash 命令。这是一种 **能力范围界定** ，使工作流变得可预测且可审计。

来源： [weather-orchestrator.md](https://zread.ai/shanraisshan/claude-code-best-practice/.claude/commands/weather-orchestrator.md#L1-L8), [weather-orchestrator.md](https://zread.ai/shanraisshan/claude-code-best-practice/.claude/commands/weather-orchestrator.md#L14-L22)

**通过 进行能力范围界定** ：限制命令的工具访问比在自然语言中编写禁止性规则更可靠。如果一个命令根本无法“看到” 工具，它就不能使用它 —— 无论提示词写得多么巧妙。将此模式应用于任何需要严格委托边界的工作流。

## 提炼出的设计原则

本演练不仅仅关于一个天气应用 —— 它是一个可扩展到任何 Claude Code 编排工作流的五个架构原则的 **参考实现** ：

1. **命令作为编排器，而非执行者** ：命令委托一切。它从不获取数据，从不渲染输出。它只负责协调。这使命令的提示词保持简短，其模型保持轻量，其行为保持可预测。
2. **预加载技能用于领域知识** ：当 Agent 需要遵循特定流程（API URL、字段路径、数据格式）时，通过 frontmatter 字段预加载这些指令。Agent 将它们视为上下文 —— 而不是要调用的工具。
3. **独立技能用于生成产物** ：当你需要一次性能力（生成 SVG、撰写报告、格式化输出）时，请使用 工具。该技能在调用者的上下文中运行，并且可以读取对话中已存在的数据。
4. **通过 进行能力范围界定** ：在配置级别限制每个组件可以执行的操作。一个无法看到 的命令就不会意外绕过你的 Agent。
5. **失效安全防护机制** ：当依赖项可能失败（API 调用、Agent 响应）时，定义明确的失败条件并停止流水线，而不是试图静默恢复。
