## 三大机制概览

每种机制在编排栈中占据不同的角色。Command 是用户触发的入口点。Agent 在隔离的上下文窗口中运行，可以自主执行。Skill 是可复用的流程，Claude 可以自动调用，或者由其他组件显式调用。下表捕获了区分它们的各个维度：

| --- | --- | --- | --- |
| **文件位置** |  |  |  |

| **用户可调用** | 始终可以 —— 通过 | 永远不行 —— 无 菜单项 | 默认可以；设置 将其隐藏 |

## 分层编排模式

典型的模式是将这三种机制串联成一个有向流水线： **Command → Agent → Skill** 。Command 作为面向用户的入口点和工作流协调器。它将数据密集型或自主性的工作委托给 Agent，而 Agent 可能会以 Skill 的形式携带预加载的领域知识。随后，Command 会内联调用额外的 Skill 以进行渲染、文件输出或其他最终处理步骤。

每一层都具有单一职责。Command 负责编排用户交互并进行委托；Agent 在隔离的上下文中自主获取数据；Skill 负责渲染输出。这种分离意味着你可以独立替换任何一层 —— 例如用 JSON 导出器替换 SVG Skill，或者更换 Agent 的数据源，而无需触碰其他部分。

## 决策框架：何时使用何种机制

机制的选择取决于三个问题： **由谁触发** 、 **是否需要隔离** 以及 **是否可复用** ？以下框架将每个问题映射到合适的机制。

### 当你需要一个用户发起的工作流入口点时，请使用 Command

Command 永远不会被自动调用 —— 它们严格存在于 前缀之后。这使得它们非常适合编排多步骤工作流，在这种场景下用户需要显式选择加入。由于 Command 的内容在触发之前不会被注入到会话上下文中，因此它们能使 Claude 会话保持精简。

天气预报编排器 Command（）完美地展示了这一点：它将自己的工具限制为 、 和 ，强制执行严格的委托契约。它甚至包含了一个 *失效安全护栏* —— 如果 Agent 没有返回数字格式的温度，步骤 3（创建 SVG）将被完全阻断。

来源： [claude-agent-command-skill.md](https://zread.ai/shanraisshan/claude-code-best-practice/reports/claude-agent-command-skill.md#L51-L62) 、 [weather-orchestrator.md](https://zread.ai/shanraisshan/claude-code-best-practice/.claude/commands/weather-orchestrator.md#L8-L20)

### 当任务需要自主执行且需要上下文隔离时，请使用 Agent

Agent 在自己的上下文窗口中运行，这意味着它们的探索和工具使用不会污染主对话。它们可以跨会话携带持久化记忆，预加载 Skill 以获取领域知识，并在受限制的工具集和权限模式下运行。

天气预报 Agent（）使用了 ，将工具限制为读取/Web 操作，预加载 作为领域知识，并将读数存储在项目范围的记忆中以便进行历史比较。该 Agent 仅返回结构化结果 —— 它永远不会直接写入输出文件。

来源： [claude-agent-command-skill.md](https://zread.ai/shanraisshan/claude-code-best-practice/reports/claude-agent-command-skill.md#L37-L50) 、 [weather-agent.md](https://zread.ai/shanraisshan/claude-code-best-practice/.claude/agents/weather-agent.md#L1-L30)

### 当你需要可复用、可自动调用的流程时，请使用 Skill

Skill 是最通用的机制。Claude 会将其 与用户意图进行匹配，并可以在不使用任何 Command 的情况下自动调用它们。它们也可以通过 Skill 工具被 Command 显式调用，或者在启动时预加载到 Agent 中。

具体的优先级规则如下：

1. **Skill 优先** —— 以零上下文开销内联运行；这始终是首选
2. **Agent 其次** —— 当 Skill 不可用（）或任务确实需要自主性和隔离时使用
3. **Command 永不自动调用** —— Command 没有自动发现途径；它们仅在用户显式输入 时触发

如果你同时禁用了 Skill 和 Agent 的模型调用（设置 并移除/弱化 Agent 的描述），Claude 将回退到其自身的通用知识 —— 根本不会触发任何扩展机制。用户必须显式输入 Command 才能触发操作。这是纯选择性加入工作流的正确模式。

## 上下文隔离策略

上下文管理是这三种机制差异最大的地方，选择错误的隔离级别是导致 Bug 的常见原因。

| 策略 | 机制 | 适用场景 |
| --- | --- | --- |
| **内联（共享）** | Command、Skill（默认） | 轻量级任务、渲染、格式化 —— 组件需要访问主对话上下文 |
| **隔离（独立窗口）** | Agent（始终） | 数据获取、探索、自主工作 —— 组件不应污染主上下文 |
| **分支（按需）** | 带有 的 Skill | 需要隔离但又不值得定义完整 Agent 的繁重转换任务 |

Agent 始终在其自己的上下文窗口中运行。这意味着 Agent 返回的数据必须通过 Agent 工具的响应显式传回 —— 它不会自动在 Command 的上下文中可用。请仔细设计你的 Agent 返回契约。

天气预报系统具体展示了这一点：Agent 在隔离环境中获取数据，以文本形式返回 ，Command 在自己的上下文中接收该文本，然后再将其传递给内联的 SVG Skill。如果你需要 SVG Skill 执行繁重的处理，可以在其前置数据中添加 —— 但对于简单的模板填充，内联是正确的选择。

- **Agent skills** （预加载）：设置 —— 它们纯粹作为领域知识，在启动时注入到 Agent 的上下文中。它们包含指令、API 端点和数据转换规则。
- **Direct skills** （调用）：设置 —— 它们出现在 菜单中，可以由 Claude 自动调用，并通过 Skill 工具从 Command 调用。

## 关键设计原则

从实际实现中总结出了四项原则，适用于你构建的任何编排流水线：

1. **每层单一职责。** Command 负责编排，Agent 负责获取，Skill 负责渲染。任何一层都不应做另一层的工作。天气预报编排器的“执行契约”通过明确的禁令来强制执行这一点。
2. **通过 Skill 预加载实现渐进式呈现。** Agent 通过预加载的 Skill 接收领域知识，而不是将其承载在提示词中。这使得 Agent 定义能够专注于工作流，并允许你独立更新知识。
3. **失效安全护栏。** 每个步骤在继续之前都应验证其输入。编排器会检查 Agent 是否返回了数字格式的温度，然后再调用 SVG Skill —— 从而防止垃圾进垃圾出的传播。
4. **模型差异化。** 使用更便宜的模型（haiku）进行编排和渲染，使用更强的模型（sonnet）进行推理至关重要的数据获取。天气预报系统在 Command 中使用 ，在 Agent 中使用 。
