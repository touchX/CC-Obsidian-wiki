[Claude Skills](https://zhida.zhihu.com/search?content_id=273815956&content_type=Article&match_order=1&q=Claude+Skills&zhida_source=entity) 该安装那些Skills呢？今天咱们就一篇文章讲清楚。

下面分三段：先讲我现在挑 skill 的三条硬线，然后是新手必装 5 个、进阶推荐 5 个，最后是几类我劝你别装的”流量陷阱”。

## 我现在挑 skill 的三条硬线

**第一条：一周用得到三次以上。**

skill 的描述会被注入每一轮对话的 [system prompt](https://zhida.zhihu.com/search?content_id=273815956&content_type=Article&match_order=1&q=system+prompt&zhida_source=entity) ， **不管你这轮用没用它** 。所以一个 skill 占的不是磁盘，是 Claude 的注意力。我以前装很多”以备不时之需”的 skill，结果三个月没碰过，但每次对话都在替它付 [token 税](https://zhida.zhihu.com/search?content_id=273815956&content_type=Article&match_order=1&q=token+%E7%A8%8E&zhida_source=entity) 。

现在我的标准简单粗暴：一周内用不到三次的，不装。

**第二条：没它，prompt 写起来真的会累。**

有些 skill 装不装无所谓——你用一段 5 行的 prompt 就能搞定的事，没必要为它专门挂个 skill。

但有些不一样。比如让 Claude 处理一份带表格、图片、批注的 [docx](https://zhida.zhihu.com/search?content_id=273815956&content_type=Article&match_order=1&q=docx&zhida_source=entity) 文件，没装 docx skill 的话，你得跟它解释半天”这个表格是合并单元格”“这段是批注不是正文”，装了之后直接读，连解释都不用。这种”省心差距”明显的，才值得占位置。

**第三条：描述精准，不会乱触发。**

这条是我吃过亏才悟出来的。有些 skill 描述写得特别宽，比如”擅长处理任何文档”，结果你写代码时它也来激活一下；”通用写作助手”，结果你发个聊天消息它也插一脚。

好的 skill 描述会清晰说明触发场景。 **装之前先看 [SKILL.md](https://zhida.zhihu.com/search?content_id=273815956&content_type=Article&match_order=1&q=SKILL.md&zhida_source=entity) 的 description 字段** ，那一行决定了它会不会污染你的 context。

## 新手必装 5 个

这五个的共同特点：通用、低门槛、装上立刻有用。如果你刚开始用 Claude Code，没装别的，先把这五个补齐。

### 1\. skill-creator——你造 skill 的入口

**它干嘛的** ：让 Claude 帮你造 skill。给你一个交互式访谈，问你”你最常做的重复工作是什么”“理想中的输出长什么样”，谈完直接生成 SKILL.md 文件给你。

**为什么必装** ：你不可能永远用别人的 skill。当你发现自己在反复跟 Claude 解释同一段 prompt（比如”用我的公众号风格写”“按这个格式输出报告”），就该把它沉淀成 skill 了。skill-creator 是这条路的起点。

**真实场景** ：我写公众号有一套固定的语气和结构要求。最早我每次都把规则贴进 CLAUDE.md，后来写多了发现，把这套规则做成 skill 之后，我在任何对话里说”按强西风格写”它就自动加载，省下每次几百字的解释。

### 2\. pdf——处理 PDF 几乎是刚需

**它干嘛的** ：读 PDF、写 PDF、合并、拆分、加水印、提取表格、填表单、OCR——所有跟 PDF 相关的操作。

**为什么必装** ：PDF 是工作里出现频率仅次于聊天截图的文件类型。合同、研究报告、产品手册、银行账单——什么都是 PDF。装它和不装它的差别是：你直接发文件让它读 vs 你得先转成图片或文本再发。

**真实场景** ：我前几天用它把一份 30 多页的合作方案 PDF 拆成单章，按章节分发给不同对接人。装之前我得自己用 PDF 软件操作，装之后一句话搞定。

### 3\. docx——读写 Word 的最佳方案

**它干嘛的** ：处理.docx 文件。包括复杂格式（表格、批注、tracked changes）、图片插入替换、查找替换、按模板生成报告。

**为什么必装** ：办公场景里 Word 文档绕不开。给老板写月度报告、给客户改合同、整理面试反馈——很多人现在还在习惯让 Claude 输出文本然后自己去 Word 里粘贴格式，装了 docx 这一步彻底省了。

**真实场景** ：上次帮朋友改一份几十页的提案，要把所有”我们公司”批量替换成新公司名、保留批注、加上 tracked changes。没 skill 的话光让 Claude 听懂这个流程就要 10 分钟，装了之后一句话。

### 4\. canvas-design——做封面图、海报、排版

**它干嘛的** ：基于设计原则生成.png 和.pdf 视觉作品。海报、信息图、封面图、说明卡片都行。

**为什么必装** ：自媒体和创作者的痛点之一是出图。Midjourney 出来的太”AI 味”，自己开 PS 太慢。canvas-design 走的是”设计原则 + 代码生成”路线——你描述要什么，它按真正的设计语言布局生成。

**真实场景** ：我公众号每篇都要一张 900×383 的封面图（你看到的开头那张就是）。以前用 Figma 自己拼，半小时起步；现在按我定义好的”深蓝渐变 + 红色竖线”模板生成，3 分钟搞定。

### 5\. xlsx 或 pptx（二选一，看你的工作侧重）

**它干嘛的** ：分别处理 Excel 和 PowerPoint。功能基本对应 docx 在 Word 上的能力——读、写、改、导出、生成图表。

**为什么二选一** ：很多人这两个都装上了，但其实大多数职业角色只重度用一个。做数据/财务/运营的，xlsx 是命；做销售/咨询/培训的，pptx 用得更多。两个都装会让你的 skill 空间塞满”以备不时之需”，违反第一条硬线。

**判断方法** ：打开你最近 30 天的工作文件，xlsx 多还是 pptx 多，装那个。

## 进阶推荐 5 个

这五个不是必需，但如果你已经把上面五个用熟了，这五个会让你的 Claude 进入”我替你干完整条工作流”的形态。

### 1\. 歸藏的 NanoBanana-PPT-Skills——pptx 不够看的时候

**它是什么** ：op7418（歸藏）开源的 PPT skill。GitHub 上 2.1K star，4 月最热的中文圈 skill 之一。和官方 pptx 不一样，它走的是”杂志风格 / 横滑 / WebGL 流体背景 / 单文件 HTML 输出”路线，更偏内容展示和发布会风格。

**为什么进阶** ：官方 pptx 给你的是”传统 PPT 文件”，能在 PowerPoint 里继续编辑。但当你要做的是分享到朋友圈的精美单页、发布会式的横滑展示、或者带动效的网页 PPT——官方那套就力不从心了。这时候歸藏这个就值得装。

**注意** ：这个和官方 pptx 是互补关系，不是替代。两个都装的话要避免描述重叠，最好把其中一个的触发条件改严。

### 2\. mcp-builder——开发者的能力外挂

**它干嘛的** ：教 Claude 怎么写 MCP server。MCP 是让 Claude 连接外部 API 的标准协议（你装的 Slack 连接、Notion 连接，背后都是 MCP）。

**为什么进阶** ：当你发现没有现成的 MCP 能连你公司内部的某个系统（CRM、报销系统、自研 API），mcp-builder 让你自己造一个。这是从”用别人的工具”到”造自己的工具”的分水岭。

**真实场景** ：之前我有一个需求是让 Claude 直接读写飞书的多维表格，开源 MCP 没有完全对得上的——我自建应用拿到的是公司 token，不是公开 API。用 mcp-builder 把它包成自己的 MCP server，问题就解决了。

### 3\. web-artifacts-builder——做交互页面/小工具

**它干嘛的** ：构建复杂的 React + Tailwind + shadcn/ui 页面。和官方那个能做单文件 HTML 的不一样，这个支持多组件、状态管理、路由——基本是个”前端项目生成器”。

**为什么进阶** ：你给团队做个内部小工具（数据看板、表单收集、计算器），以前要写好几天前端代码。现在一句话描述，它给你一个能跑的交互式工具。

**真实场景** ：我自己写过一个把 Markdown 转公众号格式的小工具（wx-formatter，单文件 HTML），就是用这套路子做出来的——给我每天写公众号的工作流省了一个反复手动复制粘贴样式的步骤。

### 4\. code-review（NeoLabHQ）——多角色 PR 审查

**它干嘛的** ：把 PR 审查拆成三个角色（bug-hunter / security / quality），分别用不同 prompt 审，最后综合输出。

**为什么进阶** ：单线程的”帮我 review 一下这段代码”经常漏东西。这个 skill 用 multi-agent 思路把审查粒度拆细——一个角色专门挑 bug，一个角色专门挑安全问题，一个角色专门看代码风格——比一个人粗糙地从头看到尾要稳。

**适用人群** ：每天要 review 多个 PR 的、独立开发者想给自己代码加层把关的、团队没人帮你 review 的。

### 5\. last30days-skill——选题灵感和热点追踪

**它干嘛的** ：抓 Reddit / X / YouTube / Hacker News 过去 30 天的热门内容，按主题聚合。

**为什么进阶** ：内容创作者最大的成本不是写，是选题。这个 skill 让你随时能问”过去 30 天 AI 编程领域最热的话题是什么”“设计师圈最近吵什么”。是给写公众号、做 newsletter、运营 X 账号的人量身定做的。

**真实场景** ：我现在每周用它扫一次”Claude 相关讨论”，比手动刷 X 高效得多。这篇文章里几个数据点（歸藏 PPT skill 的浏览量、mckaywrigley 倒戈那条）就是这么挖出来的。

## 这几类我劝你别装

最后说几类我装过、踩过坑、然后果断卸了的。不点具体名字，避免引战，但你按下面这几个特征自己对照。

![](https://pic2.zhimg.com/v2-3395deff0022417f5dedea8f5c07ef85_1440w.jpg)

**第一类：描述写得特别宽的”通用助手”型 skill。**

比如名字叫”super-writer”或”all-in-one-helper”这种，描述里一堆”擅长任何文档/任何写作/任何任务”。这种 skill 几乎一定会和你别的 skill 打架。它的触发条件太松，会在你完全不需要它的对话里被加载，污染 context。

**第二类：你每月用不到一次的”行业特化”skill。**

比如某个特定 SaaS 工具的封装、某个冷门数据库的查询包装。装之前你觉得”以后说不定用得上”，但 skill 不是收藏品。它躺着也在吃 token。等你真用得上的那天再装回来，最多多花 5 分钟。

**第三类：和官方 skill 高度重叠的”功能仿制”型 skill。**

社区里有很多自己做的 docx / pdf / xlsx 仿制品，包装得花里胡哨。但坦率说，Anthropic 官方那一套已经覆盖 90% 的场景，质量、稳定性、文档都更好。除非你有非常特殊的需求（比如歸藏 PPT 那种风格化输出），否则没必要往同一品类里塞两个。

**第四类：超过 30 天没被触发过的 skill。**

这条是我现在的固定动作：每个月翻一次 `~/.claude/skills/` ，看哪些 SKILL.md 文件的访问时间是上个月的——卸掉。装回来的成本是 0，留着的成本是每天在背景吃你的 token。

### 最后一句话

写到这儿其实你应该看出来了——挑 skill 这事，本质上不是”装哪些”，是”为你的工作流配一台专用工作站”。

桌上摆什么工具，决定了你能高效干什么活。把工作站塞满工具的人不见得效率高，反而是只留高频核心、其他用到再拿的人，速度最稳。

我自己现在的清单维持在 25 个左右——写作 5 个（公众号 / 推文 / 飞书文档 / 邮件 / 长报告各一）、办公 4 个（docx / pdf / xlsx / canvas-design）、开发 6 个（skill-creator / mcp-builder / code-review 等）、内容研究 3 个（last30days / 资料抓取 / 翻译），剩下是几个我自己写的私房 skill。

每个我都过得了上面那三条硬线。

如果你这周也想清理一下自己的 skill，可以从一件最简单的事做起：

**打开你的 ~/.claude/skills/ 目录，把上个月没访问过的全部移出去（不删，只移到一个 archive 文件夹）。**

下个月你会感谢自己。

以上，既然看到这里了，如果觉得有帮助，随手点个赞、收藏、爱心三连吧。
