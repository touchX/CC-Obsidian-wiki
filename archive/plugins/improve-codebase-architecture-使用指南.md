# improve-codebase-architecture Skill 使用指南

> **目标读者**：希望改善代码库架构、发现重构机会、整合紧密耦合模块的开发者
> **核心价值**：提升可测试性和 AI 可导航性
> **阅读时间**：完整阅读约 25 分钟，快速参考部分可随时查阅

# 第一部分：理解概念

## 为什么架构深度重要

### 核心问题：浅层架构的代价

当代码库中存在大量"浅层"模块时，你会遇到：

1. **理解成本高** — 需要在多个小模块之间来回跳转才能理解一个概念
2. **修改风险大** — 一个逻辑变更需要在多处同步修改
3. **测试困难** — 真正的 bug 隐藏在调用关系中，而非被测试的函数内部
4. **AI 辅助效果差** — AI 难以建立完整的上下文模型

### 深度架构的收益

| 收益维度 | 说明 | 实际效果 |
|---------|------|----------|
| **Leverage（杠杆率）** | 每单位接口可获得的行为量 | 调用者学习一次，获得大量能力 |
| **Locality（局部性）** | 变更、bug、知识集中在一个地方 | 修复一次，全局生效 |
| **Testability（可测试性）** | 接口即测试表面 | 测试和调用者跨过同一个 seam |
| **AI-Navigability** | AI 可以完整理解模块边界 | 更精准的代码分析和生成 |

### 深度 vs 浅浅对比

```
┌─────────────────────────────────────────────────────────────┐
│  浅层模块（Shallow Module）                                  │
├─────────────────────────────────────────────────────────────┤
│  接口复杂度 ≈ 实现复杂度                                      │
│                                                             │
│  class OrderProcessor {                                     │
│    validateOrder(order)      // 需要理解 5 个参数            │
│    calculateTax(order)       // 需要理解税务规则              │
│    applyDiscount(order)      // 需要理解折扣逻辑              │
│    checkInventory(order)     // 需要理解库存系统              │
│    processPayment(order)     // 需要理解支付网关              │
│    updateShipping(order)     // 需要理解物流接口              │
│    sendConfirmation(order)   // 需要理解通知系统              │
│  }                                                          │
│                                                             │
│  问题：调用者需要理解 7 个方法的全部细节                      │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  深层模块（Deep Module）                                     │
├─────────────────────────────────────────────────────────────┤
│  大量行为隐藏在简洁接口之后                                   │
│                                                             │
│  class OrderIntake {                                        │
│    submit(order: Order): OrderConfirmation                 │
│  }                                                          │
│                                                             │
│  // 接口只有 1 个方法，但背后隐藏了：                        │
│  // - 验证、税务、折扣、库存、支付、物流、通知               │
│  // - 所有错误处理                                          │
│  // - 所有事务管理                                          │
│  // - 所有重试逻辑                                          │
│                                                             │
│  收益：调用者只需理解"提交订单，获得确认"                     │
└─────────────────────────────────────────────────────────────┘
```

## 核心词汇图解

### 必须使用的术语（精确一致）

本 skill 要求使用精确一致的词汇。不要使用"component"、"service"、"API"、"boundary"等替代词。

```
┌──────────────────────────────────────────────────────────────┐
│                    Module（模块）                            │
├──────────────────────────────────────────────────────────────┤
│  定义：任何有接口和实现的东西                                 │
│  规模无关：函数、类、包、跨层切片都适用                        │
│  避免使用：unit, component, service                          │
└──────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────┐
│                 Interface（接口）                            │
├──────────────────────────────────────────────────────────────┤
│  定义：调用者必须知道的一切                                   │
│  包含：类型签名、不变量、顺序约束、错误模式、配置、性能特性    │
│  避免使用：API（太窄）、signature（太窄）                      │
└──────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────────────┐
│              Implementation（实现）                           │
├──────────────────────────────────────────────────────────────┤
│  定义：模块内部的代码                                         │
│  与 Adapter 的区别：Adapter 描述角色，Implementation 描述内容   │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                  Depth（深度）                               │
├──────────────────────────────────────────────────────────────┤
│  定义：接口处的杠杆率                                         │
│  深层 = 大量行为隐藏在小接口后                                │
│  浅层 = 接口复杂度接近实现复杂度                              │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                   Seam（接缝）                               │
├──────────────────────────────────────────────────────────────┤
│  定义：可以不原地编辑就改变行为的位置                         │
│  来源：Michael Feathers 的术语                               │
│  避免使用：boundary（与 DDD 的 bounded context 混淆）         │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                 Adapter（适配器）                            │
├──────────────────────────────────────────────────────────────┤
│  定义：在 seam 处满足接口的具体东西                           │
│  描述：角色（填什么槽），而非物质（里面有什么）                │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                Leverage（杠杆率）                            │
├──────────────────────────────────────────────────────────────┤
│  定义：调用者从深度获得的东西                                  │
│  效果：每单位接口学习量获得更多能力                            │
│  一次实现，在 N 个调用点和 M 个测试中回收                      │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                Locality（局部性）                             │
├──────────────────────────────────────────────────────────────┤
│  定义：维护者从深度获得的东西                                  │
│  效果：变更、bug、知识、验证集中在一个地方                     │
│  修复一次，全局生效                                           │
└──────────────────────────────────────────────────────────────┘
```

### 关系图谱

```
Module（1）──── 有 ────→ Interface（1）
    │                        │
    │                        ├── 位于 ──→ Seam（1）
    │                        │              │
    │                        │              └── 被 ──→ Adapter（N）满足
    │                        │
    ├── 产生 ──→ Leverage（对调用者）
    │
    └── 产生 ──→ Locality（对维护者）
```

### 核心原则

1. **删除测试**
   ```
   想象删除这个模块：
   - 如果复杂度消失 → 它是透传层（无用）
   - 如果复杂度重现于 N 个调用者 → 它在赚取它的位置
   ```

2. **接口即测试表面**
   ```
   调用者和测试跨过同一个 seam
   如果需要测试"越过"接口 → 模块形状可能错了
   ```

3. **一个适配器 = 假设 seam；两个适配器 = 真实 seam**
   ```
   不要引入 seam，除非至少有两个适配器（通常是生产 + 测试）
   单适配器 seam 只是间接层
   ```

# 第二部分：实践指南

## 完整工作流程

### 第 1 步：探索（Explore）

#### 1.1 读取领域上下文

```bash
# 首先读取项目的领域词汇表
cat docs/CONTEXT.md

# 读取相关 ADR（架构决策记录）
ls docs/adr/
cat docs/adr/0007-*.md  # 示例
```

**目的**：
- 使用项目的领域语言来命名好的 seam
- 避免重新讨论已记录的决策

#### 1.2 使用 Explore 代理探索代码库

```
调用 Agent 工具，subagent_type=Explore
```

**探索重点**（不要使用僵化启发式，有机探索）：

| 探索维度 | 寻找什么 | 信号 |
|---------|---------|------|
| **概念分散** | 理解一个概念需要跳转多个模块？ | 在 A、B、C、D 文件间来回切换 |
| **浅层模块** | 接口几乎和实现一样复杂？ | 公开方法 ≈ 私有方法数量 |
| **虚假可测试性** | 纯函数被提取用于测试，但真正 bug 隐藏在调用中？ | 测试通过了，但生产还是出 bug |
| **seam 泄漏** | 紧密耦合的模块跨越 seam 泄漏？ | 模块 A 需要知道模块 B 的内部结构 |
| **不可测试区域** | 代码库中未测试或难以测试的部分？ | 没有 tests/ 目录，或测试需要大量 setup |

**应用删除测试**：
```
对任何怀疑是浅层的模块：
问：删除它会集中复杂度，还是只是移动它？
"是，集中" → 你想要的信号
```

### 第 2 步：呈现候选方案（Present Candidates）

#### 2.1 候选方案格式

为每个加深机会提供：

```
编号. [简短标题]

文件：涉及的文件/模块
问题：为什么当前架构造成摩擦
解决方案：纯英文描述会改变什么
收益：用 locality 和 leverage 解释，以及测试如何改善

使用 CONTEXT.md 词汇表达领域
使用 LANGUAGE.md 词汇表达架构
```

#### 2.2 ADR 冲突处理

```
如果候选方案与现有 ADR 矛盾：
- 仅在摩擦足够值得重新访问 ADR 时提出
- 清楚标记："_与 ADR-0007 矛盾 — 但值得重新打开，因为…_"
- 不要列出每个理论重构 ADR 禁止的内容
```

#### 2.3 不要提出接口

```
在这个阶段不要提出接口
问用户："你想探索哪个候选方案？"
```

### 第 3 步：质询循环（Grilling Loop）

一旦用户选择候选方案，进入质询对话：

```
与用户一起走设计树：
  约束、依赖、加深模块的形状
  seam 后面是什么、哪些测试会存活
```

**副作用即时发生**（当决策结晶时）：

| 决策类型 | 即时行动 |
|---------|---------|
| 用 CONTEXT.md 中没有的概念命名加深模块？ | 将术语添加到 CONTEXT.md |
| 对话中澄清模糊术语？ | 立即更新 CONTEXT.md |
| 用户用负载承载原因拒绝候选方案？ | 提供 ADR："想让我记录这个为 ADR 吗？" |
| 想探索加深模块的替代接口？ | 参见 INTERFACE-DESIGN.md |

**ADR 提供条件**：
- 仅当原因对未来探索者实际需要时
- 跳过短暂原因（"现在不值得"）
- 跳过不言自明的原因

## 真实案例分析

### 案例 1：订单处理系统

#### 当前状态（浅层架构）

```
┌─────────────────────────────────────────────────────────────┐
│  问题：理解"处理订单"需要协调 7 个类                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  OrderValidator ←─ OrderCalculator ←─ OrderApplier           │
│       ↓                  ↓                  ↓               │
│  OrderChecker ←── OrderProcessor ←── OrderShipper            │
│       ↓                  ↓                  ↓               │
│           OrderNotifier                                   │
│                                                             │
│  调用者需要理解：                                            │
│  - 7 个类的接口                                              │
│  - 它们之间的调用顺序                                        │
│  - 错误如何在它们之间传播                                    │
│  - 如何在它们之间共享事务                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 深化候选方案

```
1. 订单摄入模块（Order Intake Module）

文件：order/validator.ts, order/calculator.ts, order/applier.ts,
      order/checker.ts, order/processor.ts, order/shipper.ts,
      order/notifier.ts

问题：
当前架构将"处理订单"分散在 7 个类中。
理解完整流程需要在它们之间跳转。
变更需要在多处同步。
测试需要 mock 7 个依赖。

解决方案：
合并为单一 OrderIntake 模块，接口只有：
  submit(order: Order): OrderConfirmation

实现隐藏：
  - 验证、税务、折扣、库存、支付、物流、通知
  - 所有错误处理
  - 所有事务管理
  - 所有重试逻辑

收益：
Leverage：调用者学习一个方法，获得完整订单处理能力
Locality：订单处理的所有知识集中在一个模块
测试：通过 OrderConfirmation 验证结果，不需要 mock 7 个类
```

### 案例 2：数据访问层

#### 当前状态（浅层 Repository）

```
┌─────────────────────────────────────────────────────────────┐
│  问题：Repository 接口几乎和实现一样复杂                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  interface UserRepository {                                 │
│    findById(id: string): Promise<User>                     │
│    findByEmail(email: string): Promise<User>               │
│    findByUsername(username: string): Promise<User>         │
│    findAllActive(): Promise<User[]>                        │
│    findAllInactive(): Promise<User[]>                      │
│    findAllByRole(role: Role): Promise<User[]>              │
│    findAllByCreatedAfter(date: Date): Promise<User[]>       │
│    create(user: User): Promise<User>                       │
│    update(user: User): Promise<User>                       │
│    delete(id: string): Promise<void>                       │
│    softDelete(id: string): Promise<void>                   │
│  }                                                          │
│                                                             │
│  实现（PostgresUserRepository）：                            │
│    需要实现 11 个方法                                        │
│    每个方法都有不同的 SQL 查询                              │
│    调用者需要知道用哪个方法查询什么                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 深化候选方案

```
2. 用户查询模块（User Query Module）

文件：repo/user-repository.ts, repo/user-queries.ts

问题：
Repository 接口有 11 个方法，几乎暴露所有查询变体。
调用者需要知道用哪个方法。
添加新查询类型需要修改接口。

解决方案：
合并为单一 UserQuery 模块，接口只有：
  query(spec: UserSpec): Promise<User[]>

实现隐藏：
  - 所有查询组合
  - SQL 生成逻辑
  - 查询优化

UserSpec 示例：
  { id: "123" }              // findById
  { email: "a@b.com" }       // findByEmail
  { status: "active" }       // findAllActive
  { role: "admin" }          // findAllByRole
  { createdAfter: date }     // findAllByCreatedAfter
  { email: "a@b.com", status: "active" }  // 组合查询

收益：
Leverage：调用者学习一个 query 方法和 UserSpec 结构
Locality：查询逻辑集中，易于优化
测试：测试 query 方法和 UserSpec，不需要测试 11 个方法
扩展：添加新查询不需要修改接口，只扩展 UserSpec
```

## 接口设计决策

### 何时使用替代接口设计

当用户想为选定的加深候选方案探索替代接口时，使用**并行子代理模式**。

基于"Design It Twice"（Ousterhout）原则：
> 你的第一个想法不太可能是最好的。

### 接口设计流程

#### 步骤 1：框架问题空间

在生成子代理之前，编写用户可见的问题空间解释：

```
- 任何新接口需要满足的约束
- 它将依赖的依赖，以及它们属于哪一类（见 DEEPENING.md）
- 一个粗略的说明性代码草图，使约束具体化
  （不是提案，只是使约束具体化的方式）
```

**向用户展示**，然后立即进入步骤 2。
用户在子代理并行工作时阅读和思考。

#### 步骤 2：生成子代理

使用 Agent 工具并行生成 3+ 个子代理。
每个必须为加深模块产生**根本不同**的接口。

**给每个子代理不同的技术简报和设计约束**：

| 子代理 | 设计约束 | 目标 |
|--------|---------|------|
| Agent 1 | 最小化接口 — 最多 1-3 个入口点，最大化每个入口的杠杆率 | 简洁优先 |
| Agent 2 | 最大化灵活性 — 支持多种用例和扩展 | 灵活性优先 |
| Agent 3 | 优化最常见调用者 — 使默认情况琐碎 | 便利性优先 |
| Agent 4（如适用） | 围绕 ports & adapters 设计跨 seam 依赖 | 解耦优先 |

**简报内容**：
- 文件路径
- 耦合细节
- 依赖类别（来自 DEEPENING.md）
- seam 后面有什么

**包含词汇**：
- LANGUAGE.md 词汇（module, interface, seam, adapter, leverage）
- CONTEXT.md 词汇（项目的领域语言）

**每个子代理输出**：
1. 接口（类型、方法、参数 — 加上不变量、顺序、错误模式）
2. 使用示例，展示调用者如何使用它
3. 实现在 seam 后面隐藏什么
4. 依赖策略和适配器（见 DEEPENING.md）
5. 权衡 — 哪里杠杆高，哪里杠杆低

#### 步骤 3：呈现和比较

**顺序呈现设计**，以便用户可以吸收每一个，然后用散文比较它们。

**对比维度**：
- **Depth**（接口处的杠杆率）
- **Locality**（变更集中在哪里）
- **Seam placement**（seam 放在哪里）

**给出你自己的推荐**：
你认为哪个设计最强，为什么。
如果不同设计的元素可以很好结合，提出混合方案。
**要有观点** — 用户想要强力的阅读，而不是菜单。

## 词汇速查表

| 术语 | 定义 | 避免使用 | 关系 |
|------|------|---------|------|
| **Module** | 任何有接口和实现的东西 | unit, component, service | 有 1 个 Interface |
| **Interface** | 调用者必须知道的一切 | API, signature | 位于 Seam |
| **Implementation** | 模块内部的代码 | - | - |
| **Depth** | 接口处的杠杆率 | - | Module 的属性 |
| **Seam** | 可以不原地编辑改变行为的位置 | boundary | Interface 的位置 |
| **Adapter** | 在 seam 处满足接口的具体东西 | - | 满足 Interface |
| **Leverage** | 调用者从深度获得的东西 | - | Depth 产生 |
| **Locality** | 维护者从深度获得的东西 | - | Depth 产生 |

## 依赖分类卡片

### 类别 1：进程内（In-process）

```
纯计算、内存状态、无 I/O

特点：
✅ 总是可加深
✅ 合并模块并通过新接口直接测试
✅ 不需要适配器

测试策略：
直接通过接口测试
```

### 类别 2：本地可替换（Local-substitutable）

```
有本地测试替代品（PGLite for Postgres, 内存文件系统）

特点：
⚠️ 如果替代品存在则可加深
✅ 加深模块使用替代品运行测试套件
✅ seam 是内部的；模块外部接口没有 port

测试策略：
使用替代品测试
```

### 类别 3：远程但拥有（Remote but owned）

```
跨网络边界的自己的服务（微服务、内部 API）

特点：
⚠️ 需要 Ports & Adapters
✅ 在 seam 定义 port（接口）
✅ 加深模块拥有逻辑；传输作为 adapter 注入
✅ 测试使用内存 adapter；生产使用 HTTP/gRPC/queue adapter

推荐形状：
"在 seam 定义 port，为生产实现 HTTP adapter，
为测试实现内存 adapter，使逻辑坐落在一个深层模块中，
即使它跨网络部署。"
```

### 类别 4：真正外部（True external）

```
你不控制的第三方服务（Stripe, Twilio 等）

特点：
❌ 不可加深
✅ 加深模块将外部依赖作为注入 port
✅ 测试提供 mock adapter

测试策略：
使用 mock 测试
```

## Seam 纪律

### 核心规则

```
一个适配器 = 假设 seam
两个适配器 = 真实 seam
```

**不要引入 port**，除非至少有两个适配器被证明（通常是生产 + 测试）。
单适配器 seam 只是间接层。

### 内部 vs 外部 Seam

```
深层模块可以有：
  - 内部 seams（实现私有，自己的测试使用）
  - 外部 seam（在它的接口）

不要仅仅因为测试使用内部 seams 就通过接口暴露它们
```

---

## 测试策略：替换而非分层

### 旧测试处理

```
浅层模块上的旧单元测试一旦在加深模块的接口处存在测试就变成废物 — 删除它们
```

### 新测试编写

```
在加深模块的接口处编写新测试
接口是测试表面
```

### 测试断言

```
测试通过接口断言可观察结果，而不是内部状态
测试应该存活内部重构 — 它们描述行为，而非实现
```

### 重构测试

```
如果实现改变时测试必须改变 → 它在测试越过接口
```

## 常见场景清单

### 场景 1：多个小类做一件事

**信号**：
- 理解一个概念需要在 A、B、C、D 类间跳转
- 每个类有 1-3 个方法

**行动**：
- 合并为单一深层模块
- 接口：1-3 个入口点
- 实现：隐藏所有小类的逻辑

**预期收益**：
- Leverage：调用者学习 1 个接口而非 4 个
- Locality：所有知识集中在一个地方

---

### 场景 2：Repository/Service 有 10+ 个方法

**信号**：
- 接口暴露每个查询变体
- 添加新查询需要修改接口

**行动**：
- 合并为查询模块
- 接口：`query(spec: Spec): Promise<Result[]>`
- 实现：隐藏所有查询组合

**预期收益**：
- Leverage：一个方法支持所有查询组合
- Locality：查询逻辑集中，易于优化

---

### 场景 3：纯函数提取用于测试

**信号**：
- 纯函数被提取"为了可测试"
- 真正 bug 隐藏在函数如何被调用中
- 测试通过但生产失败

**行动**：
- 识别函数的使用模式
- 将模式封装为深层模块
- 接口：描述模式，而非函数

**预期收益**：
- Leverage：调用者使用模式，不组装函数
- Locality：模式逻辑集中，bug 无处隐藏

---

### 场景 4：跨网络边界的自己的服务

**信号**：
- 微服务之间通过 HTTP/gRPC 通信
- 逻辑分散在多个服务中

**行动**：
- 定义 port（接口）在 seam
- 加深模块拥有逻辑
- 注入传输作为 adapter

**预期收益**：
- Leverage：逻辑集中，传输可替换
- Locality：业务规则在一个地方

---

### 场景 5：第三方服务集成

**信号**：
- 直接调用 Stripe、Twilio 等
- 测试需要真实凭证或 mock 到处

**行动**：
- 定义 port（接口）在外部依赖
- 加深模块将依赖作为注入 port
- 测试提供 mock adapter

**预期收益**：
- Leverage：业务逻辑与第三方解耦
- Locality：集成逻辑集中

## 快速决策树

```
开始
  │
  ├─ 发现潜在浅层模块？
  │   │
  │   ├─ 是 → 应用删除测试
  │   │   │
  │   │   ├─ 删除后复杂度消失？
  │   │   │   └─ 是 → 不是加深候选（透传层）
  │   │   │
  │   │   └─ 删除后复杂度重现于 N 个调用者？
  │   │       └─ 是 → 加深候选 ✓
  │   │
  │   └─ 否 → 继续探索
  │
  ├─ 有加深候选？
  │   │
  │   ├─ 是 → 呈现候选（不提出接口）
  │   │   │
  │   │   └─ 用户选择候选？
  │   │       │
  │   │       └─ 是 → 质询循环
  │   │           │
  │   │           ├─ 需要探索替代接口？
  │   │           │   └─ 是 → 生成 3+ 个子代理
  │   │           │       └─ 呈现和比较
  │   │           │
  │   │           └─ 用户批准？
  │   │               └─ 是 → 编写 spec
  │   │
  │   └─ 否 → 继续探索
  │
  └─ 完成
```
