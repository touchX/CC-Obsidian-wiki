# Receiving-Code-Review 技能使用完全指南

> 来源：obra/superpowers 插件 v5.0.7
> 整理：2026-05-05

---

## 概述

Code Review Reception 的核心原则：**代码审查需要技术评估，不是情感表演**。

```
★ 核心原则：
- 验证后再实施
- 假设前先问
- 技术正确性高于社会舒适度
```

---

## 响应模式

```
收到代码审查反馈时：

1. READ: 完整阅读反馈不反应
2. UNDERSTAND: 用自己的话重述需求（或问）
3. VERIFY: 对照 codebase 现实验证
4. EVALUATE: 对这个 codebase 技术上合理吗？
5. RESPOND: 技术确认或合理反驳
6. IMPLEMENT: 一次一个，测试每个
```

---

## 禁止的回应

**绝不：**
- "你说得太对了！"（明确违反 CLAUDE.md）
- "好观点！" / "优秀的反馈！"（表演性的）
- "我现在就实施那个"（验证前）

**应该：**
- 重述技术需求
- 问澄清问题
- 用技术推理反驳（如果错误）
- 直接开始工作（行动 > 话语）

---

## 处理不清晰的反馈

```
如果任何项不清楚：
  停止 - 还不实施任何东西
  对不清晰项请求澄清

为什么：项可能是相关的。 partial 理解 = 错误实施。
```

**示例：**
```
用户："修复 1-6"
你理解 1,2,3,6。不清楚 4,5。

❌ 错误：现在实施 1,2,3,6，稍后问 4,5
✅ 正确："我理解 1,2,3,6。需要先澄清 4 和 5 再继续。"
```

## 来源特定处理

### 来自用户

- **信任** - 理解后实施
- 如果范围不清楚**仍然问**
- **不表演性同意**
- **跳到行动**或技术确认

### 来自外部审查者

```
实施前：
  1. 检查：对这个 codebase 技术上正确吗？
  2. 检查：破坏现有功能吗？
  3. 检查：当前实现的原因？
  4. 检查：在所有平台/版本上工作吗？
  5. 检查：审查者理解完整上下文吗？

如果建议看起来错误：
  用技术推理反驳

如果无法轻易验证：
  说出来："没有 X 我无法验证这个。应该 [调查/问/继续]？"

如果与用户之前的决定冲突：
  停止，先与用户讨论
```

**用户的规则：** "外部反馈 - 怀疑但仔细检查"

---

## YAGNI 检查"专业"功能

```
如果审查者建议"正确实施"：
  grep codebase 实际使用

如果没有使用："这个端点没有被调用。移除它（YAGNI）？"
如果使用了：那就正确实施
```

**用户的规则：** "你和审查者都向我汇报。如果我们不需要这个功能，不要添加。"

---

## 实施顺序

```
对于多项反馈：
  1. 首先澄清任何不清晰的
  2. 然后按此顺序实施：
     - 阻塞问题（破坏、安全）
     - 简单修复（拼写错误、导入）
     - 复杂修复（重构、逻辑）
  3. 每次单独测试修复
  4. 验证无回归
```

---

## 何时反驳

当：
- 建议破坏现有功能
- 审查者缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对这个技术栈技术上不正确
- 存在遗留/兼容性原因
- 与用户架构决策冲突

**如何反驳：**
- 用技术推理，不是防御
- 问具体问题
- 引用工作的测试/代码
- 如果是架构问题涉及用户

**如果不方便大声反驳的信号：** "Strange things are afoot at the Circle K"

---

## 承认正确反馈

当反馈是正确的时：
```
✅ "Fixed. [简短描述什么改变了]"
✅ "Good catch - [具体问题]。在 [位置] 修复。"
✅ [只修复并在代码中展示]

❌ "你说得太对了！"
❌ "好观点！"
❌ "谢谢catch那个！"
❌ "谢谢 [任何]"
❌ 任何感激表达
```

**为什么不要谢谢：** 行动说明一切。只修复它。代码本身显示你听到了反馈。

**如果你发现自己要写"谢谢"：** 删除它。陈述修复。

---

## 优雅纠正你的反驳

如果你反驳错了：
```
✅ "你是对的 - 我检查了 [X] 它确实 [Y]。现在实施。"
✅ "验证了你是正确的。我最初的理解是错的因为 [原因]。正在修复。"

❌ 冗长道歉
❌ 为你的反驳辩护
❌ 过度解释
```

事实陈述修正并继续。

---

## 常见错误

| 错误 | 修复 |
|------|------|
| 表演性同意 | 陈述需求或只行动 |
| 盲目实施 | 先对 codebase 验证 |
| 不测试批量 | 一次一个，测试每个 |
| 假设审查者正确 | 检查是否破坏东西 |
| 避免反驳 | 技术正确性 > 舒适度 |
| partial 实施 | 先澄清所有项 |
| 无法验证仍然进行 | 陈述限制，请求方向 |

## 实际示例

### 表演性同意（错误）

```
审查者："移除遗留代码"
❌ "你说得太对了！让我移除那个..."
```

### 技术验证（正确）

```
审查者："移除遗留代码"
✅ "检查中... 构建目标是 10.15+，这个 API 需要 13+。需要遗留以保持向后兼容。当前实现有错误的 bundle ID - 修复还是放弃 pre-13 支持？"
```

### YAGNI（正确）

```
审查者："用数据库、日期过滤器、CSV 导出实现适当的指标跟踪"
✅ "Grepped codebase - 没有任何调用这个端点。移除它（YAGNI）？或者有我遗漏的使用？"
```

### 不清晰项（正确）

```
用户："修复 1-6"
你理解 1,2,3,6。不清楚 4,5。
✅ "理解 1,2,3,6。需要先澄清 4 和 5 再实施。"
```
