# 技术教学错误追踪系统

> **目的**：记录错误、分析根因、防止再犯
> **触发条件**：每次发现技术错误或幻觉内容

## 📝 错误记录模板

### 错误元数据

```yaml
error_id: ERR-001
date: 2026-05-02
severity: critical
category: hallucination
status: resolved
```

### 错误描述

**错误内容**：教用户使用 XML 标签语法（`<invoke>`, `<parameter>`）

**正确内容**：Command 使用纯 Markdown 自然语言

**影响范围**：创建了完全错误的教学文档

### 根因分析

| 层面 | 问题 | 根本原因 |
|------|------|----------|
| **流程** | 没有先验证 | 违反 Wiki-First 原则 |
| **认知** | 过度自信 | 凭记忆回答，没有验证 |
| **资源** | 忽视项目资料 | 项目中有正确答案但没有读取 |
| **机制** | 缺乏检查 | 没有强制性的验证流程 |

### 防止再犯措施

1. **创建强制检查清单** → `teaching-accuracy.md`
2. **建立三重验证协议** → `verification-protocol.md`
3. **实施 Wiki-First 强制规则**

### 验证检查

```
[ ] 检查清单是否已创建？✅
[ ] 验证协议是否已建立？✅
[ ] 类似错误是否可以防止？✅
```

## 🔄 持续改进机制

### 每次错误后

1. **记录错误**：使用此模板
2. **分析根因**：找到根本原因
3. **创建机制**：建立防止再犯的规则
4. **验证机制**：测试新规则是否有效

### 定期审查（每周）

```
[ ] 回顾本周错误记录
[ ] 评估现有机制有效性
[ ] 识别新的潜在风险
[ ] 更新检查清单和协议
```

## 📊 错误分类

| 类型 | 描述 | 严重程度 | 防止措施 |
|------|------|----------|----------|
| **幻觉** | 编造不存在的语法/功能 | Critical | 强制三重验证 |
| **过时** | 使用旧版本信息 | High | 检查文档日期 |
| **误解** | 理解错误但来源真实 | Medium | 多源交叉验证 |
| **遗漏** | 信息不完整 | Low | 完整性检查清单 |
